# Passing ENV variables, ARGs and Configurations

## Passing ENV variables, ARGs and Configurations

✅ Remove secrets from repo (`.gitignore` .`env.*prod`)

✅ Use a **secret manager** for credentials; **mount them or inject at runtime** (not build time).



✅ Confirm whether value is build-time (ARG) or runtime (ENV)

✅ Document required variables and defaults in `README` or sample `.env.example`

✅ Prefer `env_file` for many runtime variables

✅ Keep **non-sensitive defaults** in <mark style="color:blue;">**`.env.example`**</mark> (checked in).

✅ Store **per-environment env file** out-of-repo (or generated by CI).

✅ Use **docker-compose.yml** with <mark style="color:blue;">**`build.args`**</mark> for build-time, <mark style="color:blue;">**env\_file**</mark> for runtime.



✅ Use templating (<mark style="color:blue;">**`entrypoint`**</mark> + <mark style="color:blue;">**`envsubst`**</mark>) for complex structured configs

* **entrypoint** to render config templates from envs
* Keep a single config file inside the container (<mark style="color:blue;">**`config.yml.tpl`**</mark>)

#### `Dockerfile` — build-time args -> runtime ENV, and entrypoint usage

* **`entrypoint.sh`** — template a single config file from environment variables

{% code title="Dockerfile" %}
```docker
# Build-time args
ARG APP_VERSION
ARG API_URL

# Set runtime ENV using ARG as default
ENV APP_VERSION=${APP_VERSION:-dev}
ENV API_URL=${API_URL:-https://api.example.com}

# Install envsubst if you want to template config at container start
RUN apk add --no-cache bash gettext

COPY entrypoint.sh /usr/local/bin/entrypoint.sh
RUN chmod +x /usr/local/bin/entrypoint.sh

ENTRYPOINT ["/usr/local/bin/entrypoint.sh"]
CMD ["./start-server"]
```
{% endcode %}

{% code title="entrypoint.sh " %}
```sh
#!/usr/bin/env bash
set -e

# Render templates (use envsubst from gettext)
if [ -f /app/config.yml.tpl ]; then
  envsubst < /app/config.yml.tpl > /app/config.yml
fi

# Exec the main command
exec "$@"
```
{% endcode %}



#### `:-` syntax & `:=` syntax

The **`:-`** and **`:=`** syntax are  **default value operators** from shell variable expansion, commonly used in Docker and configuration files&#x20;

* **docker-compose(`:-` syntax)** & **shell scripts(`:=` syntax)** automatically evaluate these values as defaults if environment variables with same name are not available
* but YAML files won't automatically set them as defaults&#x20;

{% code title="docker-compose.yml" %}
```yml
version: "3.8"
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
      args:
        # Pass build-time args from environment (compose substitutes from environment or .env)
        APP_VERSION: ${APP_VERSION:-1.0.0}
        API_URL: ${API_URL}
    image: myapp:${APP_VERSION:-1.0.0}
    env_file:
      - .env                 # keep many runtime vars in one file
    environment:
      # Per-service overrides or values that need interpolation inline
      - LOG_LEVEL=${LOG_LEVEL:-info}
```
{% endcode %}

In `${APP_VERSION:-1.0.0}`:

* `APP_VERSION` is the environment variable name
* `:-` / `:=` means "use default if environment variable with the same name is unset or empty"
* `1.0.0` is the default value

**Behavior:**

* If `APP_VERSION` is set and not empty → uses the environment variable value
* If `APP_VERSION` is unset or empty → uses `1.0.0` as the default

{% code title=".env" %}
```dotenv
# Application
APP_VERSION=1.0.0
ENVIRONMENT=development

# Server
SERVER_HOST=0.0.0.0
SERVER_PORT=3000
SERVER_TIMEOUT=30000
SERVER_BODY_LIMIT=10mb
```
{% endcode %}

#### docker run/build usage (direct)

Pass many runtime envs from a file:&#x20;

```bash
docker run --env-file ./env.list -e EXTRA=val myimage
```

Pass build args when building the image:&#x20;

```shellscript
docker build --build-arg APP_VERSION=1.2.3 --build-arg API_URL=https://... -t myimage .
```



#### build-time vs runtime

**Build-time** = during docker build

* files copied into the image (e.g. a `.env` copied by Dockerfile) or ENV instructions become part of the image and are fixed in that image

**Runtime** = when you run the container with docker run or docker-compose up

* Docker/Compose injects environment variables (`environment`, `env_file`, `--env`, etc.) and mounts secrets (e.g. under /run/secrets)

<mark style="color:blue;">**Precedence**</mark>&#x20;

* Values set by environment: in **docker-compose.yml** or **docker run --env** have higher precedence than env\_file
* Secrets that **entrypoint.sh** reads and exports at container start will be present in the environment for envsubst and can override image defaults
* Image-baked `ENV` or a <kbd>.env</kbd> file copied into the image are **lowest precedence** — they act as defaults only if no runtime value is provided.

<mark style="color:blue;">**How envsubst fits**</mark>

* **`envsubst`** substitutes variables using the container's current environment at the moment it runs (i.e. runtime environment)
* If **entrypoint.sh** sets fallback defaults (for example `ENVIRONMENT=${ENVIRONMENT:-development}`) before calling envsubst, those defaults are used when no runtime value was supplied
* If a variable is not set at runtime and no fallback was exported, envsubst will substitute an empty string (unless your script sets defaults first)



## Handling secrets properly

* Never put passwords, API keys, or private certs in committed .env files or in Dockerfile ARGs (they may end up in image layers or VCS)
*   Use:

    * <mark style="color:blue;">**`Docker secrets (Swarm)`**</mark> or <mark style="color:blue;">**`compose v3.1+ secrets`**</mark>, or
    * **Bind-mount a file** with restricted permissions, or
    * **External secret manager** (Vault/AWS Secrets Manager/GCP Secret Manager)&#x20;

    and read at runtime.

{% code title="entrypoint-with-secrets.sh" %}
```sh
#!/usr/bin/env sh
set -e

# NEVER template secrets into files
# Instead: export them as ENV for app to read directly

if [ -f /run/secrets/db_password ]; then
  export DB_PASSWORD=$(cat /run/secrets/db_password)
fi

if [ -f /run/secrets/api_key ]; then
  export API_KEY=$(cat /run/secrets/api_key)
fi

# Template NON-SECRET config only
envsubst '$DB_HOST $DB_PORT $DB_NAME' < /app/config.yml.template > /app/config.yml

# App reads DB_PASSWORD from process.env, not config file
exec "$@"
```
{% endcode %}

#### ❗How ENV variables actually persist; Get backed into the Image

1. Anything you put in an `ENV` instruction or `ARG` instruction stays in the image metadata forever, since they are used at image build time.
2. ✅ But environment variables set via an `ENTRYPOINT` instruction is set at runtime, so they don't get baked into the image layer.

❌<br>

#### **Docker BuildKit Secret Mounts** ✅ (Recommended)

{% hint style="info" %}
This is the **industry standard** for handling secrets during build.
{% endhint %}

{% code title="Dockerfile" %}
```docker
# syntax=docker/dockerfile:1

FROM node:18 AS builder
# Mount secret at build time (never persisted in layers)
RUN --mount=type=secret,id=npm_token \
    NPM_TOKEN=$(cat /run/secrets/npm_token) && \
    npm config set //registry.npmjs.org/:_authToken=$NPM_TOKEN && \
    npm install

FROM node:18 AS runtime
COPY --from=builder /app/node_modules ./node_modules
# Secrets are NOT in this layer
```
{% endcode %}

Build with:

{% code title="bash" %}
```bash
docker build --secret id=npm_token,src=.npm_token .
```
{% endcode %}

**Benefits:**

* Secrets never stored in image layers
* Can't be extracted from final image
* Secure and auditable



## Handling Complex Configurations

> <mark style="color:green;">**Configuration file templating + Layered config**</mark>
>
> 🏅 gold standard for complex Docker deployments

Recommendation

1. **Use approach 1** (entrypoint templating with `envsubst`)
2. **Structure**:
   * `config.yml.template` in repo (checked in)
   * `.env.dev`, `.env.staging`, `.env.prod` (gitignored, or `.env.example` checked in)
   * Secrets via Docker secrets or mounted files
3. **Compose file** references `env_file` per environment
4. **Validation** in entrypoint catches errors before app starts

### Implementation Approaches

#### 🥇 Approach 1: Entrypoint templating (recommended)

**Best for**: Most production scenarios, 12-factor apps

{% code title="Dockerfile" %}
```dockerfile
FROM node:18-alpine

# Install envsubst for templating
RUN apk add --no-cache gettext

WORKDIR /app
COPY package*.json ./
RUN npm ci --production

# Copy config template and entrypoint
COPY config.yml.template /app/config.yml.template
COPY entrypoint.sh /usr/local/bin/
RUN chmod +x /usr/local/bin/entrypoint.sh

COPY . .

# ENTRYPOINT is the script that runs on container start up, not supposed to run @ build time
# Thus, place it in the last stage of multi-stage docker build
ENTRYPOINT ["/usr/local/bin/entrypoint.sh"] 
CMD ["node", "server.js"]
```
{% endcode %}

{% code title="config.yml.template" %}
```yml
database:
  host: ${DB_HOST}
  port: ${DB_PORT}
  name: ${DB_NAME}
  pool:
    min: ${DB_POOL_MIN}
    max: ${DB_POOL_MAX}

redis:
  hosts:
    - ${REDIS_HOST_1}
    - ${REDIS_HOST_2}
  cluster: ${REDIS_CLUSTER_MODE}

api:
  rate_limit:
    enabled: ${RATE_LIMIT_ENABLED}
    requests_per_minute: ${RATE_LIMIT_RPM}

  level: ${LOG_LEVEL}
  format: ${LOG_FORMAT}
  outputs:
    - type: stdout
    - type: file
      path: ${LOG_FILE_PATH}

features:
  new_ui: ${FEATURE_NEW_UI}
  beta_api: ${FEATURE_BETA_API}
```
{% endcode %}

{% code title="entrypoint.sh" %}
```sh
#!/usr/bin/env sh
set -e

echo "Rendering configuration from environment..."

# Render main config
envsubst < /app/config.yml.template > /app/config.yml

# Validate config exists and is not empty
if [ ! -s /app/config.yml ]; then
  echo "ERROR: Configuration rendering failed"
  exit 1
fi

echo "Configuration ready"

# Execute the main command
exec "$@"
```
{% endcode %}

**Best for**: Microservices with service discovery, dynamic config updates

#### 🥉 Approach 3: Application-native config loading

**Best for**: Apps with mature config libraries (Viper, dotenv + validation)

**Pros**: No templating step, native to app language\
**Cons**: Config logic spread across code, harder to audit final config

{% code title="config/config.go" %}
```go
package config

import (
    "github.com/spf13/viper"
)

type Config struct {
    Database DatabaseConfig
    Redis    RedisConfig
    API      APIConfig
}

func Load() (*Config, error) {
    // 1. Set defaults
    viper.SetDefault("database.port", 5432)
    viper.SetDefault("redis.cluster", true)
    
    // 2. Read config file
    viper.SetConfigName("config")
    viper.SetConfigType("yaml")
    viper.AddConfigPath("/app")
    viper.AddConfigPath(".")
    
    if err := viper.ReadInConfig(); err != nil {
        // Config file not required
    }
    
    // 3. Override with environment variables
    viper.SetEnvPrefix("APP")
    viper.AutomaticEnv()
    
    var cfg Config
    if err := viper.Unmarshal(&cfg); err != nil {
        return nil, err
    }
    
    return &cfg, cfg.Validate()
}
```
{% endcode %}

{% code title="config.yml" %}
```yml

  port: 5432

  hosts: []
```
{% endcode %}

### Advanced: Layered configuration strategy

**Production-grade pattern**: Merge multiple config sources

{% code title="entrypoint.sh" %}
```sh
#!/usr/bin/env sh
set -e

CONFIG_DIR=/app/config
FINAL_CONFIG=/app/config.yml

# Layer 1: Base defaults (baked into image)
cp ${CONFIG_DIR}/base.yml ${FINAL_CONFIG}

# Layer 2: Environment-specific (mounted or baked)
if [ -f ${CONFIG_DIR}/${ENVIRONMENT}.yml ]; then
  yq eval-all 'select(fileIndex == 0) * select(fileIndex == 1)' \
    ${FINAL_CONFIG} ${CONFIG_DIR}/${ENVIRONMENT}.yml > /tmp/merged.yml
  mv /tmp/merged.yml ${FINAL_CONFIG}
fi

# Layer 3: Runtime overrides from ENV vars
envsubst < ${FINAL_CONFIG} > /tmp/final.yml
mv /tmp/final.yml ${FINAL_CONFIG}

# Layer 4: Secrets (injected, never templated)
if [ -f /run/secrets/db_password ]; then
  export DB_PASSWORD=$(cat /run/secrets/db_password)
fi

exec "$@"
```
{% endcode %}

**Config layers**:

1. `config/base.yml` — defaults, checked into git
2. `config/production.yml` — per-environment structure, checked in
3. Environment variables — runtime values (hosts, ports, flags)
4. Secrets — injected at runtime, never in config files



### Validation & debugging

{% code title="entrypoint-with-validation.sh" %}
```sh
#!/usr/bin/env sh
set -e

echo "=== Rendering configuration ==="
envsubst < /app/config.yml.template > /app/config.yml

echo "=== Validating configuration ==="
# Use yq, jq, or app-specific validator
yq eval /app/config.yml > /dev/null || {
  echo "ERROR: Invalid YAML syntax"
  exit 1
}

# Optional: schema validation
if command -v ajv > /dev/null; then
  ajv validate -s /app/config.schema.json -d /app/config.yml
fi

# Debug mode: print sanitized config
if [ "$DEBUG_CONFIG" = "true" ]; then
  echo "=== Final configuration (secrets redacted) ==="
  yq eval '.database.password = "***" | .api.key = "***"' /app/config.yml
fi

exec "$@"
```
